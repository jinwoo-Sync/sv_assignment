# 좋은 코드란 무엇인가

저는 좋은 코드란 결국 **동료를 위한 코드**라고 생각합니다.

---

## 협업 관점 — 동료가 읽을 수 있는 코드

실제로 개발을 해보면 코드는 혼자 작성하고 끝나는 게 아니라, 여러 사람이 계속 이어서 수정하고 발전시키는 경우가 훨씬 많습니다. 개발자뿐만 아니라 QA나 다른 팀과는 명세서나 문서를 기준으로 소통하지만, 개발자들 간에는 코드 자체가 가장 직접적인 **소통 수단**이 됩니다. 그래서 단순히 프로그램이나 기능이 "동작한다"는 것만으로는 부족하고, 다른 사람이 봐도 바로 이해하고 이어서 작업할 수 있는 상태가 중요하다고 생각합니다.

예를 들어, 변수명이 직관적이지 않거나 코드 스타일이 들쭉날쭉하거나, 예외 처리를 제대로 하지 않은 경우를 보면 처음에는 빨리 끝난 것 같지만 나중에 반드시 문제가 됩니다. 특히 협업 환경에서는 이런 부분들이 쌓이면서 디버깅이나 수정에 훨씬 더 많은 시간이 들어가고, 결국 전체 개발 속도를 떨어뜨리는 경험을 했습니다.

저는 코드를 하나의 **개발자 간 소통 방식**이라고 생각합니다. 말을 하지 않아도 코드 자체만으로 의도와 흐름이 전달되어야 한다고 생각합니다. 그래서 네이밍이나 구조를 최대한 일관되게 유지하고, 읽는 사람이 흐름을 자연스럽게 따라갈 수 있는 코드를 만들려고 노력합니다.

---

## 기능 관점 — 테스트할 수 있는 코드

기능 시퀀스는 문서나 명세를 보고 구현할 수 있습니다. 하지만 **프로그램의 안정성을 보장하는 책임**은 문서가 아니라 코드 구조 자체에서 나옵니다. 아무리 기능이 문서대로 동작해도, 코드가 잘게 쪼개져 있지 않으면 단위 테스트도 통합 테스트도 작성하기 어렵고, 결국 그 기능이 얼마나 안정적으로 동작하는지 검증할 방법이 없어집니다.

그래서 저는 기능 관점에서 좋은 코드란 **테스트할 수 있는 단위로 쪼개진 코드**라고 생각합니다. 그리고 그 쪼개는 데에도 일관된 규칙이 있어야 합니다. 규칙 없이 쪼개면 각자 다른 기준으로 작성된 코드가 쌓이게 되고, 누가 대신 개발하거나 함께 개발할 때 기능 구현 방식 자체가 달라져 버립니다.

잘게 쪼개고 규칙이 있는 코드는 단위 테스트와 통합 테스트를 가능하게 하고, 테스트가 쌓이면서 **코드에 신뢰성이 생깁니다.** 그 신뢰성이 곧 프로그램의 안정성이 되고, 결국 사용자가 믿고 쓸 수 있는 제품으로 이어진다고 생각합니다.

---

결국 좋은 코드는 동료가 읽을 수 있고, 테스트로 검증할 수 있으며, **신뢰가 쌓여가는 코드**라고 생각합니다.
